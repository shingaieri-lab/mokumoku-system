# マルチテナント化（会社ごとにデータを分ける）検討メモ

作成日：2026-05-20

---

## 背景・課題

現在のシステムは **「1つの営業チーム専用（シングルテナント）」** として設計されている。
複数の会社に販売・提供しようとすると、以下の問題がある。

### 現状の構造的な問題点

| 問題 | 内容 |
|---|---|
| テナントIDがない | データベースのどのデータにも `companyId` が存在しない |
| データが分離されていない | ログインすれば全リード・全アカウント・全設定にアクセスできる |
| 設定が1セットしかない | Zoho・AI（Gemini）・Googleカレンダーの設定が1組織分しか持てない |
| 管理者権限がグローバル | 管理者は全ユーザーのAPIキーやトークンにアクセスできる |

---

## 対応アプローチ

### 案① インスタンスを会社ごとに分ける（推奨・現実的）

**概念図：**
```
A社 → Vercelプロジェクト (mokumoku-a) + KVデータベース (kv-a)
B社 → Vercelプロジェクト (mokumoku-b) + KVデータベース (kv-b)
C社 → Vercelプロジェクト (mokumoku-c) + KVデータベース (kv-c)
```

**セットアップ手順（1社追加するたびに行う）：**
1. Vercelで新しいプロジェクトを作成（GitHubリポジトリをフォーク or 同じリポジトリを再リンク）
2. Vercel KV（データベース）を新規作成して接続
3. 環境変数（`KV_URL`, `ENCRYPTION_KEY` など）を会社ごとに設定
4. 独自ドメイン（例：`a-sha.mokumoku.dandori.work`）を設定
5. 管理者アカウントを作成して渡す

**メリット：**
- 今のコードをほぼ変えずに対応できる
- データが物理的に完全分離されるのでセキュリティリスクが低い
- 1社の障害が他社に影響しない
- 会社ごとに独自設定（Zoho連携・AI設定など）がしやすい

**デメリット：**
- 会社が増えるたびに手作業でセットアップが必要（自動化しないと運用コストが増える）
- 全社への機能追加・バグ修正は全プロジェクトに個別デプロイが必要
- 利用状況・データを横断的に見ることができない

**コスト感：**
- Vercelの無料プラン：プロジェクト数に上限なし（ただしKVの無料枠は1DBのみ）
- 本格運用はVercel Proプラン（$20/月〜）が現実的
- 会社が増えるごとにKV代金が加算される（従量課金）

---

### 案② 1つのシステムに `companyId` で分ける（本格SaaS・開発コスト大）

**概念図：**
```
1つのVercelプロジェクト
└── KVデータベース（1つ）
    ├── leads:{companyId}:{leadId}  ← A社のリード
    ├── leads:{companyId}:{leadId}  ← B社のリード
    ├── accounts:{companyId}        ← A社のユーザー一覧
    └── accounts:{companyId}        ← B社のユーザー一覧
```

**必要な主な実装変更：**

1. **アカウントに `companyId` を追加**
   ```javascript
   // 現在
   { id, name, email, role, ... }

   // 変更後
   { id, companyId, name, email, role, ... }
   ```

2. **全APIに `companyId` フィルタを追加**
   ```javascript
   // 現在（全リードを返す）
   const leads = await readData('leads');

   // 変更後（自社のリードのみ返す）
   const leads = await readData(`leads:${req.companyId}`);
   ```

3. **セッションに `companyId` を紐づける**
   ```javascript
   // ログイン時にcompanyIdもセッションに保存
   session = { accountId, companyId }
   ```

4. **会社登録・テナント管理の仕組みを追加**
   - 会社新規登録フロー
   - スーパー管理者（全社を管理できる）とテナント管理者を分離

5. **KVのキー設計を全面変更**
   - 現在：`leads`, `accounts`, `master_settings`
   - 変更後：`leads:{companyId}`, `accounts:{companyId}`, `settings:{companyId}`

**メリット：**
- 1つのシステムで全社を管理できる
- 機能追加・バグ修正が1回のデプロイで全社に反映される
- 利用状況を横断的に集計・分析できる
- スケールしやすい

**デメリット：**
- 実装変更の範囲が非常に大きい（全API・全データ構造の改修）
- バグがあると全社に影響する
- セキュリティ設計をミスるとデータ漏洩リスク

**開発工数の目安：**
- APIとデータ構造の改修：2〜3週間
- テナント管理画面：1〜2週間
- テスト・検証：1週間
- 合計：**1〜1.5ヶ月程度**

---

## どちらを選ぶか

| 観点 | 案① インスタンス分離 | 案② companyId分離 |
|---|---|---|
| 開発コスト | 低い（ほぼ0） | 高い（1〜1.5ヶ月） |
| 運用コスト | 会社数に比例して増える | 一定（スケールしやすい） |
| セキュリティ | 高い（物理分離） | 設計次第 |
| 会社数が少ない（〜10社） | ◎ 向いている | △ コスト過剰 |
| 会社数が多い（10社〜） | △ 運用が大変 | ◎ 向いている |
| 最初の販売・実証 | ◎ すぐ始められる | △ 時間がかかる |

### 推奨戦略

```
フェーズ1（現在〜契約10社まで）
  → 案①（インスタンス分離）でスタート
  → セットアップを半自動化するスクリプトを作ると楽になる

フェーズ2（10社を超えたら）
  → 案②（companyId分離）に移行
  → フェーズ1で得た実際の使われ方をもとに設計する
```

## 価格設計

### 原価の把握（まず押さえる）

1社あたりにかかるランニングコスト：

| 項目 | 目安 |
|---|---|
| Vercel Pro（1プロジェクト分） | $20/月 ÷ 社数（複数社で按分） |
| Vercel KV（Redis・1社1DB） | 無料枠あり。超えると $0.2/10万リクエスト程度 |
| Gemini API（AI解析） | 無料枠あり。使い方次第でほぼ$0〜数百円/月 |
| 合計（目安） | **1社あたり500〜2,000円/月** |

### 価格プラン案

#### プランA：シンプル月額制（おすすめ）

| プラン | 月額 | 対象 | ユーザー数 |
|---|---|---|---|
| スタータープラン | 9,800円/月 | 小規模チーム | 〜5名 |
| スタンダードプラン | 19,800円/月 | 中規模チーム | 〜15名 |
| プロプラン | 29,800円/月 | 大規模チーム | 〜30名 |

- 年払い割引（2ヶ月分無料）を設けると解約率が下がりやすい
- 初月無料トライアルをつけると導入ハードルが下がる

#### プランB：初期費用 + 月額制

| 項目 | 金額 |
|---|---|
| 初期セットアップ費 | 50,000〜100,000円（1回のみ） |
| 月額利用料 | 9,800〜19,800円/月 |

- セットアップ作業（Zoho連携・マスターデータ入力・研修）を含めた形で請求できる
- SaaS感よりも「導入支援込みのシステム」として売りやすい

#### プランC：成果報酬 or 従量課金（将来的な検討）

- 登録リード数や商談化数に応じた課金
- 今は計測の仕組みがないので将来フェーズ向け

### 価格設定のポイント

- **Vercelコストは利用料に含めて転嫁**（別途請求すると煩雑になる）
- **競合SaaS（Salesforce・HubSpot）が月数万円〜**なので、1〜3万円は十分安い水準
- **最初の数社は特別価格で入れて実績を作る**（5,000円/月などのモニター価格）
- 解約時のデータエクスポート手段を用意しておくと導入判断されやすい

## セットアップ自動化

### 現状（手作業でかかる時間の目安）

| 作業 | 手作業での時間 |
|---|---|
| Vercelプロジェクト作成 | 10分 |
| KVデータベース作成・接続 | 10分 |
| 環境変数の設定（10項目程度） | 15分 |
| ドメイン設定 | 10分 |
| 管理者アカウント作成 | 5分 |
| Zoho連携の初期設定案内 | 30分 |
| **合計** | **約1.5時間/社** |

10社になると15時間。自動化は必須。

### 自動化の段階

#### ステップ1：セットアップスクリプトを作る（工数：半日）

```bash
# 例：新しいテナントを追加するスクリプト
node scripts/create-tenant.js --company "A株式会社" --slug "company-a" --admin-email "admin@example.com"
```

スクリプトがやること：
1. Vercel CLI で新プロジェクトを作成
2. KV データベースを作成して紐づけ
3. `ENCRYPTION_KEY` を自動生成して環境変数にセット
4. カスタムドメインを設定
5. 初期管理者アカウントをKVに書き込む
6. 完了後に「ログインURL・初期パスワード」をターミナルに出力

#### ステップ2：管理ダッシュボードを作る（工数：1〜2週間）

```
自社用の管理画面（顧客に見せないもの）
├── テナント一覧（会社名・登録日・最終ログイン日）
├── 新規テナント追加フォーム
├── 各テナントのURL・初期パスワードリセット
└── 月次利用状況（ログイン数・リード数など）
```

#### ステップ3：セルフオンボーディング（将来フェーズ）

顧客が自分でサインアップできるページ。今は手動運用でOK。

### 優先度

```
今すぐ → ステップ1のスクリプト（半日で作れる）
3ヶ月以内 → ステップ2の管理ダッシュボード
10社超えたら → ステップ3 or 案②（companyId分離）に移行検討
```

## サポート体制

### サポート範囲の定義（最初に決めておく）

#### 対応する（サービス範囲内）

- システムが動かない・表示されないなどの不具合対応
- 操作方法の案内（使い方がわからない）
- Zoho連携の初期設定サポート
- 機能追加・カスタマイズの相談受付（別途見積もり）

#### 対応しない（スコープ外）

- Zoho CRM本体の操作サポート（Zoho社のサポートを案内）
- Google・GmailなどのGoogleサービスのサポート
- パソコン・ネットワーク環境の問題
- 顧客のデータ入力代行

### 連絡窓口の案

#### 案A：チャットツールで受ける（おすすめ・初期段階）

- **Slackの共有チャンネル**（dandori ↔ 顧客）を1社1チャンネル作る
- 問い合わせ → Slackで受ける → 対応 → 解決
- 顧客も使い慣れているので心理的ハードルが低い
- デメリット：チャンネルが増えると管理が煩雑になる（10社超えたら見直し）

#### 案B：メール窓口を作る

- `support@dandori-work.com` などの専用メールアドレスを作る
- 問い合わせをメールで受け付けて対応
- チケット管理ツール（Notionやスプレッドシートで管理）

#### 案C：問い合わせフォーム（将来フェーズ）

- Webフォームから問い合わせ → メール通知 → 対応
- Notion・Googleフォームなどで実装できる

### 対応時間とSLA（サービスレベルの約束）

| 項目 | 内容（案） |
|---|---|
| 対応時間 | 平日 10:00〜18:00 |
| 初回返信目標 | 当日中（翌営業日） |
| 重大障害（ログイン不可など） | 2時間以内に状況報告 |
| 機能要望の回答 | 1週間以内に対応可否を回答 |

### 初期オンボーディングのフロー（1社入れるときの流れ）

```
1. 契約締結
   ↓
2. セットアップ（スクリプト実行・ドメイン設定）：半日
   ↓
3. キックオフMTG（Zoom・1時間）
   - システムの使い方レクチャー
   - Zoho連携の設定
   - マスターデータ（ステータス・担当者）の初期入力
   ↓
4. 試用期間（2週間）
   - Slackで質問を随時受け付け
   ↓
5. 本稼働
   - 月次で使用状況の確認MTG（任意）
```

### サポートコストを下げるために

- **操作マニュアル（動画 or Notion）を作っておく**（同じ質問に何度も答えない）
- **よくある質問（FAQ）ページを作る**（Notionで十分）
- **Zohoの初期設定はキックオフMTGで一緒にやる**（後からの質問を減らす）
