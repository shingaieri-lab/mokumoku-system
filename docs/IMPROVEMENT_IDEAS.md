# 改善アイデア

## 🥇 優先度 高｜テストを書く

### 今どういう状態？

`lib/` の中にある関数（日付の計算・AI結果の整形・営業日のずらし処理など）は、
テストなしで動かしています。

バグを見つける方法が「実際に使ってみる」か「コードレビューで気づく」しかないため、
見落としが起きやすい状態です。

今回のセッションでも以下のバグが手作業のレビューで発見されました：

- 深夜0〜9時に前日の日付になるUTC/JSTのずれ（`AIPage`・`LeadsPage`）
- `readOnly` ユーザーに商談日が表示されない（`DealActionBar`）
- 呼び出されていない関数がそのまま残っていた（`handleCreateZohoDeal`）

### どう直す？

`lib/` の関数に対してユニットテスト（単体テスト）を書く。
ユニットテストとは「この関数にこの値を渡したらこの結果が返るはず」を自動で確認する仕組みです。

```
例：
normalizeDate("2026/5/7") → "2026-05-07" になることを自動チェック
addBizDays("2026-05-07", 3) → 土日祝をスキップして3営業日後の日付になることを自動チェック
```

テストを書いておくと、機能を変更したときに「他の部分が壊れていないか」を
コマンド1つで確認できるようになります。

### 工数の目安

- 環境構築（Vitest の導入）：2〜3時間
- `lib/date.js`・`lib/ai.js`・`lib/format.js` の主要関数にテストを書く：半日〜1日
- 費用対効果：**高**（一度書いたら永続的に働いてくれる）

## 🥈 優先度 中｜`window.__appData` を React の仕組みに置き換える

### 今どういう状態？

Zoho認証の状態・AIの設定・ログインユーザー情報などが、`window.__appData` という
「ブラウザのグローバル変数」に詰め込まれています。

```js
// コード中にこういう書き方が40箇所以上ある
window.__appData?.zohoAuthenticated
window.__appData?.aiConfig
window.__appData?.zohoConfig?.clientId
```

これ自体は動きますが、以下の問題があります：

1. **Reactが変化を検知できない**：認証状態が変わっても自動で画面が更新されない
2. **どこから読んでいるか追いにくい**：検索しないと全体像がわからない
3. **テストが書けない**：`window` オブジェクトを前提とした設計はテスト環境で動かない

### どう直す？

React Context（リアクト コンテキスト）に置き換える。
Context とは「どのコンポーネントからでも読み書きできる共有の棚」のようなものです。

```
現状：  コンポーネント → window.__appData.xxx を直接参照
改善後：コンポーネント → AppContext.xxx を参照（Reactが変化を管理）
```

### 工数の目安

- Context ファイルの作成・既存コードの置き換え：1〜2日
- 費用対効果：**中**（今は困っていないが、機能追加のたびに少しずつ複雑になっていく）

## 🥉 優先度 低｜リスト描画を最適化する

### 今どういう状態？

`LeadsPage`（リード一覧）では、検索欄に1文字入力するたびに
全リードに対してフィルター＋ソート処理が実行されています。

```
現在の流れ：
入力 → 全リードをフィルター → ソート → 描画  （毎回フルスキャン）
```

現在はリードが100件未満なので体感できませんが、1,000件を超えると
入力のたびに画面がカクつく可能性があります。

### どう直す？

`useMemo`（ユーズメモ）という React の機能を使う。
「フィルター条件が変わったときだけ再計算する」ように指示できます。

```js
// 現状（毎回計算）
const list = leads.filter(...).sort(...)

// 改善後（条件が変わったときだけ計算）
const list = useMemo(() => leads.filter(...).sort(...), [leads, fStatuses, fSource, ...])
```

### 工数の目安

- 修正自体：30分〜1時間
- 費用対効果：**低**（今は不要。リードが500件を超えてから対応しても十分間に合う）

## まとめ

| 優先度 | 内容 | 工数目安 | いつやるか |
|--------|------|----------|------------|
| 🥇 高 | ユニットテストを書く | 半日〜1日 | できるだけ早め |
| 🥈 中 | `window.__appData` を React Context に | 1〜2日 | 機能追加が落ち着いたら |
| 🥉 低 | リスト描画を `useMemo` で最適化 | 30分 | リードが500件を超えたら |

最初に取り組むなら **テストから** がおすすめです。
工数が少なく、その後の開発全体の安心感が上がります。
