# アウトバウンド共有機能 設計案

作成日：2026-05-20

---

## 概要

業務委託先（外部スタッフ）がアウトバウンド架電・メールを行い、結果を記録できる機能。
アポ獲得時はChatworkにワンボタンで報告できる。

**業務委託先がやること：架電・メールのみ**
ISチームの既存データ（リード管理・ダッシュボード等）は一切見えない。

---

## 画面構成

### メニュー分離

```
【ISチーム用メニュー（既存）】      【アウトバウンド用メニュー（新規）】
├── ダッシュボード                  └── アウトバウンド管理
├── リード管理                          ├── 架電リスト（リスト切替付き）
├── AI架電アシスト                      └── アポ獲得一覧
├── カレンダー
└── 設定
```

ログインしたロールによって表示メニューを切り替える。

---

### 画面①：架電リスト（リスト切替付き）

**リスト切替バー（上部に配置）：**
```
［5月Aリスト▼］  ［+ 新規インポート］  ［削除］
  250件 / 未架電:180 アポ:12 お断り:30 完了:28
```

- 複数のリストをインポートして切替できる
- リスト名はインポート時に任意で設定できる（例：「5月Aリスト」「6月テックリスト」）
- **リストの削除はadmin / memberが可能**（業務委託先は不可）
- 削除しても他のリストには影響しない

**架電リスト本体：**
- 1行ずつ架電・メール結果を記録できる
- 進捗サマリー（未架電 / 架電済 / アポ獲得 / お断り）をリストごとに表示

**架電ステータスの選択肢：**
- 未架電
- 不在
- 折り返し待ち
- 担当者不在
- お断り
- アポ獲得 ← 選択するとアポ情報入力ボタンが出る
- NG（今後連絡不要）

**1行に記録できる情報：**
- 架電日時
- 架電結果（上記ステータスから選択）
- メモ（自由記入）
- 連絡手段：電話 / メール

## データ構造

### KVキー設計（容量対策含む）

```
outbound_lists              → リスト一覧（名前・作成日・件数のみ。軽量）
outbound_leads:{listId}     → リストごとに別キーで保存（← 重要）
outbound_config             → Chatwork API Token・Room ID
```

**ポイント：リストごとに別キー**
- 現在の `leads` は全リードを1つのキーに入れているが、アウトバウンドは **リストID単位で分割保存** する
- 理由：1リストあたりのデータ量が多くなっても他のリストに影響しない。削除もキーごと消すだけで完結する

### 容量の目安

| 条件 | 概算サイズ |
|---|---|
| 500件・架電履歴なし | 約100KB |
| 500件・1件あたり3回の架電履歴 | 約250KB |
| 1,000件・1件あたり5回の架電履歴 | 約700KB |
| 2,000件・1件あたり5回の架電履歴 | 約1.4MB ⚠️ |

**制約：** Vercel KV（Upstash Redis）は1回の読み書きが **約1MB** が目安。
- 500〜1,000件規模なら問題なし
- 2,000件超 or 架電履歴が大量に積み重なると上限に近づく可能性がある

**対策（1,000件超の場合）：**
- リストを分割してインポートする（例：1リスト500件×2）
- 使い終わったリストは削除して容量を解放する
- 将来的に件数が増える場合は別途ストレージ設計を見直す

## 権限設計

| ロール | 既存機能 | アウトバウンド機能 |
|---|---|---|
| admin | 全操作 | 全閲覧 + リスト削除 + Chatwork設定 |
| member | 通常操作 | 全閲覧 + リスト削除 |
| **outbound**（新規） | 見えない | 架電リスト操作・アポ情報入力・Chatwork送信（リスト削除は不可） |

- 業務委託先には `outbound` ロールのアカウントを1つ発行する
- ISチームのリード・設定・AI機能は一切見えない
- **リスト削除はISチーム（admin / member）のみ可能**
