## このツールは何をするもの？

営業チームが「どの会社に連絡したか」「次はいつ連絡するか」を管理するツール。
AIが通話メモを読んで次の行動を提案したり、Googleカレンダーの空き時間を自動で探したりできる。

## ❌ 未対応：今後対応が必要なもの

### 🔴 高優先度

#### モバイル対応の不完全さ（メール・AI・設定ページ）

`isMobile` フラグでレイアウトを切り替えているが、一部の画面でカバーが不完全。

- AIページ左パネル：`width:380`（固定）→ モバイルで全幅を圧迫
- メールテンプレート：`gridTemplateColumns:"240px 1fr"`（固定）
- 設定ページのタブ：`width:"60vw"`（固定）→ モバイルでオーバーフローの可能性

**改善案：** ブレークポイントで固定幅をすべて可変に。ボタンパディングをモバイルで拡大。

---

### 🟠 中優先度

#### 削除・編集ボタンが初期状態で非常に薄い（設定ページ）

`opacity:0.3` で表示され、ホバー時だけ `opacity:1` になる。タッチデバイスではホバーが発生しないため、ボタンの存在に気づけない。

**改善案：** 常時 `opacity:0.7` 以上に変更。

#### 成功メッセージの消失が速すぎる（全体）

「✅ 下書き保存済」が3秒、「保存しました ✓」が2秒で自動消失。読む前に消えるケースがある。

**改善案：** 5〜8秒に延長、または ✕ ボタンで手動 Close。

#### エラーメッセージの視認性が低い（AIページ・フォーム全般）

エラー背景が薄く（`#ef444416`）、複数エラー時は一括 `alert()` で表示されるため、どのフィールドに問題があるか分かりにくい。

**改善案：** 背景を濃くしアイコンを付ける。各フィールドに個別エラーを表示。

#### ボタンが小さすぎてタッチ操作しにくい（テンプレート・設定リスト）

`padding:"3px 10px"`, `fontSize:12` で高さ約20px。WCAG 推奨の最小タッチ領域は 44×44px。

**改善案：** 最低 `padding:"6px 12px"`（高さ32px以上）に統一。

#### バルク操作の完了通知がない（リード管理）

複数リードのインポート・削除完了時に視覚的フィードバックがなく、処理結果が分かりにくい。

**改善案：** 画面右下にトースト通知（例：「10件追加しました ✓」）を実装。

#### ローディング表示が統一されていない（全体）

AI解析中はスピナーが表示されるが、Gmail保存・Googleタスク作成などはテキスト変化のみ。

**改善案：** 全ての非同期処理で同じスピナーを使用。

---

### 🟡 低優先度（余裕があれば）

#### AIページ：解析後の導線が不明

解析→保存した後、次に何をすればいいかの案内がない。

```
現状： 解析する → 保存完了！ ← ここで終わり

改善案：解析する → 保存完了！
                 → 「📧 フォローメールを送る」ボタン
                 → 「📅 商談候補日を探す」ボタン
```

#### 確認ダイアログがブラウザのポップアップ

削除確認に `window.confirm`（ブラウザ標準のグレーのポップアップ）を使用。
アプリのデザインと統一感がなく、ユーザーが戸惑う。

**改善案：** アプリ内モーダルで確認を取る。

#### 単一HTMLファイルの分割（長期対応）

全ロジック・スタイル・テンプレートが1ファイルに集中しており、機能追加・バグ修正の難易度が高い。
長期的にはコンポーネントごとにファイル分割を検討する。

## ✅ P0対応済み（2026-04-03 セキュリティ修正）

コミット `3d36fb0` にてP0全件を対応。

| 問題 | 対応内容 |
|------|---------|
| CSRFトークン未実装 | セッションをSameSite=Strict HttpOnly Cookieに移行。CSRFトークン不要な設計に変更 |
| 暗号化キーのフォールバックが脆弱 | フォールバック値を削除。`PASSWORD_ENC_KEY`未設定時はサーバー起動失敗させる |
| GeminiのAPIキーがURLに露出 | `/api/ai/analyze`エンドポイントを新設。APIキーはサーバー内部のみで使用 |
| Zoho Client Secretがフロントエンドに保存 | `/api/data`・ログインレスポンスからClientSecretを除外済み。設定変更時のみ送信 |
| セッショントークンがlocalStorageに保存 | HttpOnly+Secure+SameSite=Strict Cookieに移行。x-session-tokenヘッダー廃止 |
| Zoho webhookに認証がない | `ZOHO_WEBHOOK_TOKEN`によるクエリパラメータ検証を追加（Zoho側URL更新が別途必要） |
| postMessageのoriginチェックなし | `"*"`→サーバーのオリジンに変更。フロント側も受信時にorigin検証を追加 |
| Open Redirect（redirectUri） | `redirectUri`をサーバー側で自動生成に固定。設定画面は読み取り専用に変更 |

あわせてP2の「セッション有効期限が30日」もこのタイミングで7日に短縮済み。

## 🆕 新規発見：追加対応が必要な問題（2026-03-31 厳格レビュー）

### 🔴 P1 - 高優先度（バグ・機能不全）

#### レート制限がKV障害時に無効化される（`server.js:44-51`）

```javascript
try {
  const count = await kv.incr(key);
  ...
} catch { /* KVエラー時はスルー */ }
next();  // 障害時もリクエストが通過する
```

- **影響**: 外部サービス障害時にレート制限が完全に無効化され、ブルートフォース攻撃が通る
- **対策**: KV障害時は安全側に倒す（429を返す、またはリクエストをブロック）

#### Zoho webhookの同時処理でデータロスが発生（`server.js:799`）

複数のwebhookが同時に到着した場合、それぞれが全データを読み込んで上書きするため、後から書いた方が先の変更を消す（Last-Write-Wins問題）。

- **影響**: 複数のZohoイベントが同時処理されるとリードデータが失われる
- **対策**: 更新処理にロック機構またはアトミック操作を導入

#### Zohoトークンリフレッシュの競合状態（`server.js:410`）

複数のリクエストが同時にトークン期限切れを検出し、全てが`refreshZohoToken()`を呼び出す。

- **影響**: 短時間に複数回のトークンリフレッシュが発生、古いトークンを使い続ける処理が混在する可能性
- **対策**: リフレッシュ中フラグを設け、進行中は待機させる

#### API失敗時のロールバックがない（`index.html:64-69`）

```javascript
window.__appData.leads = leads;  // メモリ上はすぐ更新
apiPost('/api/leads', leads);    // 戻り値確認なし・失敗時の復元なし
```

- **影響**: ネットワークエラーなどでAPIが失敗しても、画面上は保存済みに見える。再読み込みすると変更が消える
- **対策**: `apiPost` の結果を確認し、失敗時はメモリ上のデータを元に戻す

#### 招待コードがr.okチェックなしで処理される（`index.html:4820`）

```javascript
const r = await fetch('/api/invite', {...});
const data = await r.json();  // 4xx/5xxでも実行される
setInviteCode(data.code);
```

- **影響**: エラーレスポンス時も `r.json()` を呼び出し、`undefined` のコードがセットされる
- **対策**: `if (!r.ok) throw new Error(...)` を追加

### 🟠 P2 - 中優先度

#### セッション有効期限が30日と長すぎる（`server.js:122`）

```javascript
await kv.set('session:' + token, id, { ex: 60 * 60 * 24 * 30 });
```

- **影響**: デバイス紛失・盗難時に30日間アクセスが可能な状態が続く
- **対策**: 7日以下に短縮。または定期的なセッション更新を実装

#### ページネーション未実装（`index.html:35`）

全リードデータを `window.__appData` にまるごと保持。リード数が増えると全件メモリ展開される。

- **影響**: リード数が1000件を超えると、読み込み・保存・描画すべてで遅延・フリーズが発生
- **対策**: サーバー側ページネーション、またはバーチャルスクロールの実装

#### パスワードをハッシュと暗号化の両方で保存（`server.js:156-159`）

```javascript
password: hashedPassword,     // bcrypt（復号不可）
password_enc: encryptedPassword,  // AES-256（復号可能）
```

ログイン認証にはbcryptで十分なはずで、`password_enc`（復号可能な暗号化）を保存する設計意図が不明。

- **影響**: `password_enc` が漏洩した場合、元のパスワードが復元される
- **対策**: `password_enc` の必要性を再検討。管理者がパスワードを確認する要件がないならば削除

#### Google祝日APIへの外部依存（`index.html:219`）

```javascript
const res = await fetch('https://holidays-jp.github.io/api/v1/date.json');
```

起動時に外部APIを呼び出している。失敗すると祝日判定が空のまま動作する。

- **影響**: ネットワーク制限環境・外部サービス障害時に祝日が全て無視される
- **対策**: エラー時はローカルのJP_HOLIDAYSにフォールバックするよう修正

#### エラーメッセージに技術用語が含まれる（`index.html:873`）

「ブラウザのCORSエラーが原因の場合はローカルサーバー経由（http://localhost）でアクセスしてください」

- **影響**: 一般ユーザーにはCORSの意味が不明で、対処できない
- **対策**: 「接続できませんでした。管理者にお問い合わせください」などに変更

#### OAuthエラーハンドリングのコード重複（`index.html:277` と `index.html:955`）

`redirect_uri_mismatch` などのOAuthエラー判定が2か所で全く同じ内容で繰り返されている。

- **対策**: 共通関数に切り出す

#### `ref.current` を `useEffect` の依存配列に含む（`index.html:402`）

```javascript
React.useEffect(()=>{
  ...
  return () => { document.removeEventListener("mousedown", h); };
}, [ref.current]);  // ref.current はオブジェクトの参照先が変わることがある
```

- **影響**: 特定の条件でイベントリスナーが二重登録され、クリックイベントが二重発火する可能性
- **対策**: 依存配列を `[]` にし、`ref` オブジェクト自体を渡す
